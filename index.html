A transaction is a unit of program execution that accesses and or possibly updates various data items. Acid properties of a transaction are :
Atomicity --- either transaction executes in entirety or none executes at all
Consistency --- state of database remains same both before & after transaction despite system changes
Isolation ---- despite concurrent execution of transactions , each transaction proceeds independently of others despite system changes
Durability ----- any modifications made to database as a result of transaction persist even if there are system changes

e.gs we have a schedule involving 2 accounts a and b, initial values in a and b are $ 1000 and $ 2000.
A transaction occurs that transfers $ 50 from a ->b , as a result of which , final values In a and b becomes $ 950 and $ 2050.
Clearly a consistent state as sum a +b is preserved both before and after transaction.
We now , suppose that a hd failure occurs after write – a committed but before write –b committed ,
so, now final values in a and b becomes $ 950 and $ 2000 , clearly inconsistent state,
as sum a +b is no more preserved both before and after transaction. Even if system failure did not occurs
and transaction proceeded to completion, this inconsistent state would not have been visible except during occurrence of transaction.
This again reflects atomicity . isolation is preserved. Durability is preserved through recovery mgmt. component of transactions
